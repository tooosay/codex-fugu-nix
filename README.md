# codex-fugu-flake

This is unoficial SakanaAI's fugu nix package.

The package comes with a pinned OpenAI Codex binary with the SakanaAI Fugu Codex profile.

This package has two parts:
- `pkgs/codex-fugu`: a pure package that only writes to `$out`
- `modules/home-manager.nix`: a thin Home Manager module that installs `~/.codex/fugu.json`, adds the Sakana provider block to `~/.codex/config.toml`

Fugu profile files are not vendored in this repository. They are taken from the `fugu-src` flake input:

```nix
fugu-src = {
  url = "github:SakanaAI/fugu";
  flake = false;
};
```
## Installation
```nix
inputs.codex-fugu.url = "path:/etc/nixos/modules/codex-fugu-nix";
inputs.codex-fugu.inputs.nixpkgs.follows = "nixpkgs";
```
### Home Manager (NixOS Integration)

**`home.nix`**
```nix
{osConfig, ...}:
{
  imports = [ inputs.codex-fugu.homeModules.default ];

  programs.codex-fugu = {
    enable = true;
    apiKeyFile = osConfig.sops.secrets."sakana-api-key".path;
  };
}
```
**`example-sops.nix`**
```nix
{
  sops.age.keyFile = "/home/user/**/age/keys.txt";
  sops.secrets."sakana-api-key" = {
    sopsFile = ./secrets/codex-fugu.yaml;
    key = "codex_fugu/SAKANA_API_KEY";
    owner = "user";
    mode = "0400";
  };
}
```

### Package only

```nix
environment.systemPackages = [
  inputs.codex-fugu.packages.${pkgs.system}.default
];
```

The package installs:

```text
$out/bin/codex
$out/bin/codex-fugu
$out/share/codex-fugu/fugu.json
$out/share/codex-fugu/fugu.config.toml
$out/share/codex-fugu/model_providers.sakana.toml
```

Note that `fugu.config.toml` still contains the upstream `{{CODEX_HOME}}`
template token. Rendering it is user-specific and is done by the Home
Manager module.


## Secrets

`apiKeyFile` is intentionally a string option, not a Nix path option. This avoids accidentally copying a local secret file into the Nix store.

Good:

```nix
apiKeyFile = config.sops.secrets.sakana-api-key.path;
```

Bad:

```nix
apiKeyFile = ./sakana-api-key;
apiKeyFile = "${./sakana-api-key}";
apiKeyFile = pkgs.writeText "sakana-api-key" "fish_...";
```

Recommended with [sops-nix](https://github.com/mic92/sops-nix):

```nix
{ config, ... }: {
  sops.secrets.sakana-api-key = {
    sopsFile = ./secrets/sakana.yaml;
    mode = "0400";
  };

  programs.codex-fugu = {
    enable = true;
    apiKeyFile = config.sops.secrets.sakana-api-key.path;
  };
}
```

The module also asserts that `apiKeyFile` does not point into `/nix/store`.

## What the Home Manager module deploys

Upstream `install.sh` deploys three things for Codex >= 0.134.0 ("modern"
config format), and this module mirrors all
three:

1. `~/.codex/fugu.json` — the model catalog (installed verbatim; it has no
   template tokens)
2. `~/.codex/fugu.config.toml` — the profile file that makes `codex -p fugu`
   resolve, with `{{CODEX_HOME}}` rendered to the absolute Codex home at
   build time (upstream does the same substitution in `render_template`)
3. the `[model_providers.sakana]` block injected into `~/.codex/config.toml`
   between managed markers

Without (2) the `fugu` profile simply does not exist and `codex -p fugu`
fails. The legacy (< 0.134.0) format, which injects `[profiles.fugu]` into
`config.toml` instead, is intentionally not supported.

## Override Codex

By default the package fetches the upstream Codex release binary defined in codex-pin.nix.

To use nixpkgs' Codex instead:

```nix
inputs.codex-fugu.packages.${pkgs.system}.default.override {
  codexPackage = pkgs.codex;
}
```

Or in an overlay:

```nix
(final: prev: {
  codex-fugu = prev.codex-fugu.override {
    codexPackage = prev.codex;
  };
})
```

## Codex pin and hashes

The bundled Codex version and release hashes live in:

```text
pkgs/codex-fugu/codex-pin.nix
```

Update them from the locked `fugu-src` revision with:

```bash
./scripts/update-codex-pin.sh
```

The script reads upstream `BUNDLE_CODEX_VERSION`, prefetches the matching Codex release assets for the supported Linux targets, and rewrites `codex-pin.nix`. 

## Customizing the fugu profile
`fugu.config.toml` is generated by merging your `profileSettings` over the
upstream profile (TOML-structure-aware, user wins):

```nix
programs.codex-fugu.profileSettings = {
  model_reasoning_effort = "medium";
  features.image_generation = true;
};
```

`fugu.config.toml` is deployed as a writable regular file (not a store
symlink), because Codex itself writes to the active profile file — e.g.
persisting a `/model` choice. On each activation the module performs a
three-way merge: your runtime deviations from the previous generation
(`~/.codex/.fugu.config.toml.baseline`) are treated as intent and re-applied
on top of the freshly generated profile. So `/model` choices survive both
switches and upstream config updates; keys you never touched keep following
upstream + `profileSettings`. Reverting a key back to its baseline value
returns it to declarative control. To fully reset, delete
`fugu.config.toml` and switch again.

For full manual ownership of the file, set `manageProfileConfig = false`.

If you previously ran the upstream installer, `~/.codex/fugu.json` and
`~/.codex/fugu.config.toml` already exist as regular files and Home Manager
will refuse to clobber them. Remove them (or use `home.file.<...>.force`)
before the first activation.


## Why use `fugu-src` instead of vendoring `fugu.json`?

`fugu.json` is upstream model profile data, not a user-editable config. Keeping it as a flake input makes its source and revision explicit while avoiding a local copy. The exact upstream revision is pinned in `flake.lock`.

The package still must not write to `~/.codex`; that is the Home Manager module's job.
